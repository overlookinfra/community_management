So, litmus is the tool for running acceptance tests (as opposed to the unit/integration tests, which are all passing, thankfully!). In this case, it looks like the test failure you're getting is from the `spec\acceptance\parameter_target\negative\negative_acl_spec.rb` test file, erroring on an unhandled attempt to call `.dacl` on the security descriptor which is nil at the point it's called: https://github.com/puppetlabs/puppetlabs-acl/blob/2c8ef6e716cc429a1bda2808c939d0299c2252f6/lib/puppet/provider/acl/windows.rb#L201 I _suspect_ this is because with your change to `get_security_descriptor` the SD is never being assembled in this test case (as the file does not exist). https://github.com/puppetlabs/puppetlabs-acl/pull/248/files#diff-187cf29bc01bdc6e2eee55dc03923fedb2556c21e73a511cd7d023436094a440R569 What you may want to do instead is to investigate dropping the guard altogether (since the rescue in that codeblock handles when the file doesn't exist) and see how that changes things. We have some [documentation on using Litmus to test changes on our blog](), but here's how I set up the dev environment and ran the tests locally - unfortunately, at this time, you cannot use the PDK to run acceptance tests and need to use a system ruby. I use the `RubyInstaller` PowerShell module (which leverages chocolatey and uru) for this myself, so these instructions will too. In an elevated PowerShell terminal: ```powershell # This section eases installation of ruby with dev tools on windows, which we need Install-Module RubyInstaller # This command is interactive; select ruby 2.5 or ruby 2.7 for your system architecture. Install-Ruby # In my case, I installed ruby 2.7 uru 2.7 cd /path/to/puppetlabs-acl # remove Gemfile.lock if it exists Remove-Item .\Gemfile.lock -ErrorAction SilentlyContinue # Install the needed gems bundle install --path .bundle/gems # Provision a vagrant box for running tests, then install the agent and module bundle exec rake litmus:provision_list[vagrant] bundle exec rake litmus:install_agent # You'll need to run this step again anytime you modify the module bundle exec rake litmus:install_module # You can either open up ./spec/fixtures/litmus_inventory.yaml # or run this code block - it depends on having a convertfrom-yaml function # I use the powershell-yaml module Get-Content .\spec\fixtures\litmus_inventory.yaml -Raw | ConvertFrom-Yaml | Select-Object -ExpandProperty Groups | Where-Object -FilterScript { $_.name -match 'winrm' } | Select-Object -ExpandProperty targets | Select-Object -ExpandProperty uri # Either way, you want the output of that - an IP with a port - and set it as TARGET_HOST # The value here is for the vagrant node that spun up on my machine, yours will be different $Env:TARGET_HOST = '172.19.179.52:5985' # You can run just the one test file that was failing: bundle exec rspec spec\acceptance\parameter_target\negative\negative_acl_spec.rb # You can also run the full suite: bundle exec rspec spec\acceptance # Or bundle exec rake litmus:acceptance:172.19.179.52:5985 # This module's vagrant provisioner only spins up one target to test against, # but if you wanted to run against multiple you could always do: bundle exec rake litmus:acceptance:parallel ``` If you ever want to pair on this, I'm happy to make a bit of time to help out! michaeltlombardi 17 3 false false
